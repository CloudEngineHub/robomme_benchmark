# 视频生成性能分析报告

## 问题定位

通过分析 `app-v2.py` 中的 `save_video` 函数和 `execute_step` 函数，发现以下性能瓶颈：

## 主要性能瓶颈

### 1. **VP9编码速度慢** ⚠️ 最严重
- **位置**: `app-v2.py:58` - `cv2.VideoWriter_fourcc(*'vp09')`
- **问题**: VP9是一个高质量但编码速度非常慢的编码格式，特别不适合实时视频生成
- **影响**: VP9编码速度比H.264慢5-10倍，比MPEG-4慢3-5倍

### 2. **串行处理两个视频** ⚠️ 严重
- **位置**: `app-v2.py:200-201`
  ```python
  base_video_path = save_video(session.base_frames, suffix="base")
  wrist_video_path = save_video(session.wrist_frames, suffix="wrist")
  ```
- **问题**: 两个视频串行生成，没有并行处理
- **影响**: 如果每个视频需要2秒，总共需要4秒；并行处理可以缩短到2秒

### 3. **逐帧颜色空间转换** ⚠️ 中等
- **位置**: `app-v2.py:63` - `cv2.cvtColor(frame, cv2.COLOR_RGB2BGR)`
- **问题**: 在循环中对每一帧进行RGB到BGR转换
- **影响**: 对于100帧的视频，需要100次转换操作

### 4. **帧预处理效率低** ⚠️ 轻微
- **位置**: `app-v2.py:37-45`
- **问题**: 先创建processed_frames列表，再写入，占用额外内存
- **影响**: 对于大视频，内存占用较高

### 5. **没有视频缓存机制** ⚠️ 轻微
- **问题**: 每次执行动作都重新生成视频，即使帧没有变化
- **影响**: 重复计算浪费资源

## 性能估算

假设有100帧，每帧255x255x3：

| 操作 | 耗时估算 |
|------|---------|
| VP9编码 | ~3-5秒 |
| H.264编码 | ~0.5-1秒 |
| MPEG-4编码 | ~0.3-0.6秒 |
| 帧预处理 | ~0.1-0.2秒 |
| RGB2BGR转换 | ~0.05-0.1秒 |
| **VP9总耗时** | **~3-5秒/视频** |
| **H.264总耗时** | **~0.5-1秒/视频** |

两个视频串行处理：VP9需要6-10秒，H.264需要1-2秒

## 优化建议

### 优先级1: 更换编码格式（最重要）
- 将VP9改为H.264 (`'avc1'` 或 `'mp4v'`)
- 或使用MPEG-4 (`'mp4v'`)
- **预期提升**: 速度提升5-10倍

### 优先级2: 并行处理
- 使用 `concurrent.futures.ThreadPoolExecutor` 并行生成两个视频
- **预期提升**: 时间减半

### 优先级3: 优化颜色转换
- 批量转换或使用更高效的方法
- **预期提升**: 减少10-20%时间

### 优先级4: 添加缓存
- 如果帧没有变化，复用之前的视频
- **预期提升**: 避免重复计算
